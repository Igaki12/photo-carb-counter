# Photo Carb Counter リサーチアプリ計画

## 概要
GitHub Pages 上で `docs/` から公開する React + Vite のシングルページ・リサーチプロトタイプを構築する。このアプリは、手の参照が見える 1 枚のアップロード済み食品写真から炭水化物量を推定する。ローカルでの MediaPipe Hands、食品候補ランキングのための Gemini Embedding 2、可食重量推定のための Gemini 3.5 Flash 構造化出力、そして栄養ソースとして日本食品標準成分表の炭水化物ワークブックを使用する。

デフォルトの決定事項:
- 出力は炭水化物推定値のみとし、インスリン投与量は出力しない。
- Gemini API キーは毎セッション入力し、React のメモリ内のみに保持する。
- ワークブックの全 1101 食品を候補として使用する。

## 主要な変更
- GitHub Pages 向けに設定した React + Vite アプリをスキャフォールドする:
  - `vite.config.ts`: `base: "/photo-carb-counter/"`, `build.outDir: "docs"`。
  - ビルド後に `docs/.nojekyll` を追加する。
  - アプリのソースを `docs/` の外に保持する。`docs/` はビルド出力専用とする。
- クライアントワークフローを追加する:
  - API キー入力と接続テスト。
  - 画像アップロードとプレビュー。
  - オーバーレイ付きの MediaPipe Hand Landmarker 検出。
  - 手サイズセレクター: small / standard / large。
  - アップロード画像に対する Gemini 画像埋め込み呼び出し。
  - 事前計算済み食品名埋め込みに対するドット積ランキング。
  - ユーザー選択可能な食品ランキングと手動補正。
  - LLM の重量推定実行中に食品固有の追加質問を行う。
  - 炭水化物範囲、整数の可食重量推定、短い根拠、固定の医療上の注意文を含む最終結果。
- データパイプラインを追加する:
  - `日本食品標準成分表-炭水化物編_2023.xlsx` を、food id、group、food number、name、notes、`CHOAVL` g/100g、数値パースフラグ、ソースメタデータを含むアプリ JSON に変換する。
  - `GEMINI_API_KEY` を必須とする `scripts/generate-food-embeddings.mjs` を追加する。静的読み込み用に、次元削減した正規化ベクトル（推奨 768）を出力する。
  - 埋め込みが欠落していてもアプリはビルド可能とし、明確な「埋め込みデータ未生成」状態を表示する。
- 質問ルールを追加する:
  - 食品グループ/名前の正規表現に基づくルールベース質問バンク。
  - 飲料では砂糖、シロップ、ミルク、提供量を質問する。
  - パン/サンドイッチではパンの種類/枚数、スプレッド、チーズ、ベーコン/肉、ソース/ドレッシング、甘い付け合わせを質問する。
  - 米/麺/菓子/果物ではカテゴリ固有の分量と追加物に関する質問を行う。
- 重量推定に Gemini の構造化出力を使用する:
  - 複合食品では構成要素レベルの整数グラムを優先する。
  - スキーマには、選択食品、可視構成要素、推定可食グラム、最小/最大グラム、信頼度、短い根拠を含める。
  - 最終炭水化物計算では `component grams * CHOAVL / 100` を使い、これにユーザー回答調整を加える。

## インターフェース
- `FoodItem`: `id`, `foodNo`, `group`, `name`, `carbAvailableGPer100g`, `carbMonosaccharideEqGPer100g`, `note`, `isEstimated`, `source`。
- `FoodEmbedding`: `foodId`, `vector`, `model`, `dimensionality`, `normalized`。
- `QuestionAnswer`: `questionId`, `label`, `value`, `unit`, `carbAdjustmentG?`。
- `WeightEstimate`: 構成要素の整数重量、最小/最大範囲、信頼度、根拠。
- `CarbEstimate`: 合計グラム、最小/最大グラム、選択食品/構成要素、固定注意文。

## テスト計画
- ワークブックパースのユニットテスト:
  - 括弧付き値は `isEstimated` を伴う数値としてパースされる。
  - `Tr`、`-`、空欄値が一貫して処理される。
  - コーヒー、砂糖、ミルク、パンの行が存在する。
- 炭水化物計算のユニットテスト:
  - 単一食品: grams x g/100g。
  - 複合サンドイッチ: 構成要素の合計。
  - 砂糖/ミルク回答の調整。
- ランキングのユニットテスト:
  - 正規化ドット積の並び順。
  - 手動選択した食品がランキングを上書きする。
- ブラウザ QA:
  - デスクトップおよびモバイルのレスポンシブワークフロー。
  - API キー未入力、不正 API キー、手未検出、埋め込みなし、LLM 失敗。
  - ビルドが成功し、`docs/` が静的 GitHub Pages 出力として配信可能であること。

## 前提
- これは研究用参照ツールのままとし、インスリン投与量の推奨は行ってはならない。
- API キーは localStorage、sessionStorage、IndexedDB、URL パラメータに保存しない。
- MediaPipe モデルアセットはバンドルするか、明示的な public パスから読み込む。安定しない「latest」URL からは取得しない。
- 食品埋め込みには、有効な Gemini API キーを用いた別ステップでの生成が必要である。
- アプリは推定が誤っている可能性を表示し、治療判断には医療専門家の指導が必要であることを示す。
